Analysing Success

Analysis of Common Patterns among Successful GitHub Projects.

Get Started

Abstract

Open source projects are the backbone of today's technology . We find them in programming languages that are at the core of any software (e.g. Luna or Rust), frameworks that shape the web (e.g. Laravel, React or Angular), toolkits that help scientists handle and analyse data (e.g. numpy, pandas) or development tools used by all software teams around the world (e.g. git), to name a few examples.


Do successful open source projects have common patterns?

Through this project, we hope to gain some insights into what contributes to the success of an open source project. This might allow us to identify common patterns among successful projects , and formulate some guidelines that might be beneficial to other projects' success.

Are there time patterns to when contributors submit their changes?

Day of the week?

From the above plots, it can be seen that **the majority of projects are most active during weekdays, with a sharp drop during the weekends**. However, there are certain exceptions to this rule: `pandas-dev/pandas`, `numpy/numpy`, `pallets/flask`. This makes sense, as most people rest during the weekends.

Hour of the day?

From the above graph, it can be noticed that **all projects follow a similar daily time distribution**, where the activity increases to **reach a peak, then decreases constantly and hits a minimum for 6 to 8 hours** or so.

This corresponds to **most of the commits happening during the working period**, with the least activity happening during the evening and at night. This makes sense for the following reasons: 1. For **team-driven projects**, a lot of the work will be happening during the **work hours** 2. For **other projects** and contributors in general, the contributors will also be following **daily schedules** (i.e. they sleep at night).

While projects will have contributors from all around the world, the majority is likely to be from specific countries (e.g. China, India and the US are respectively ranked 1st, 2nd and 3rd in terms of Internet users, [source](https://en.wikipedia.org/wiki/List_of_countries_by_number_of_Internet_users)) or specific to that project (e.g. `NervJS/nerv` seems to be driven by a Chinese community).

Considering the obtained results: on **average** for all 24 projects, issues take **57 days (stddev=40)** to be resolved. This is highly variable from one project to another, with some projects being much faster in resolving issues (a couple of weeks), and other beings much slower (a few months). The big range of the resolution time is likely due to the nature of the project and its issues, some projects/issues being inherently more complex than others. This means that issues require more time to properly debug and fix. Still, the projects are responsive in most cases, with the 2nd quartile (50%) of average issue resolution time being a couple of months on average. This is acceptable considering the amount of time required to inquire about the issue, debug it, fix it and test the final outcome. Thus, these results provide **another common feature** among these **successful projects** and that is in relation to **the average time required to resolve issues**. On average, most issues are resolved within a couple of months, which is a good indicator of a project's activity and responsiveness.

Can sentiment analysis on the commit messages / Issues provide any insights?

In order to obtain insights relating to the communication between the team members, as well as with the community, three different data features were analyzed: 1. Git commit messages for each project. 2. The body of each project's GitHub issues. 3. The comments of each GitHub issue per project.

Sentiment Analysis: Commit Messages

Analyzing commit messages provides insight into inter-project members communication, since commit messages are about explaining what each patch does, and are used to describe it for the other project members and for reference (e.g. when checking the history).

Sentiment Analysis: Body of Git Issues

Analyzing the body of GitHub issues provides insight into the community's tone towards a project, since issues are mainly opened by the community (i.e. users) to report bugs, request features or offer patches, and the body of an issue is where all of this is described.

Sentiment Analysis: Git Issue Comments

The results of the sentiment analysis in commit messages, GitHub issues and comments show that the neural sentiment is highly dominating. But there are still some noticeable differences between these three aspects. The commit messages are mostly full of neutral sentiments (on average ~97.5%). This is expected since commit messages have a descriptive and informative function (to provide concisely and objectively the commit's purpose). While the GitHub issues also have mostly neutral sentiments (on average ~99.5%), the amount of positive sentiments on GitHub (on average ~4.6% with stddev ~1.5%) is significantly higher than the negative ones (on average ~0.5% with stddev ~0.5%). This result could indicate a more positive and motivated community that is answering the issues, which might be related to the success.


Thus, these results provide **another common feature** among these **successful projects** and that is in relation to the **sentiment in their online communication**. Generally the neutral sentiment is highly dominating in commit messages and GitHub issues; when replying to the GitHub issues in the form of **GitHub comments**, the **sentiment tends to be more positive than negative (~4.6% vs ~0.5%)**. The type of communication seems to play an important role in these community-driven project. An objective (neutral) and non-negative tone helps enhance the communication between the community and project members, as well as between project developers themselves. This seems to be a common feature for all of these successful projects.

Can we relate the success / popularity to a responsive / helpful community (e.g. StackOverflow / Reddit subreddits)?

Several **online community** are available for **developers to learn and share their programming knowledge**. The aim of this portion of the project is to try and **access whether the success** of a project can be related to how **responsive / helpful its online community is**. In this section, **two** main communities were chosen to be analyzed, these being **StackOverflow** and **Reddit** (subreddits).

Popularity on StackOverflow

The obtained result solidifies the point made by the first question (refer to 6.1) that **some projects are more popular than others**. This also further illustrates the need to normalize in order to be able to compare with respect to certain features, something done throughout the project.

Percentage of resolved questions per project

Considering the obtained results: on **average** for these projects, **42% (stddev=13%)** of StackOverflow questions have an accepted answer. These results provide **another common feature** among these **successful projects** and that is in relation to their community **dealing with StackOverflow questions**. On average, 42% of questions are resolved, which is a relatively good indicator of a project's activity for two reasons: 1. The project is popular enough to have enough people asking questions about it, which leads to the creation of a tag about it on StackOverflow in the first place. 2. The community is active and resposive enough to answer around half of the questions in a satisfactory manner. (Other questions might also be answered, but the poster didn't select an accepted answer in those cases.)

Average Rsolution Time of StackOverflow Questions

Considering the obtained results: on **average** for these projects, it takes **216 hours (stddev=112)** for a StackOverflow question to receive a satisfactory answer (meaning it will be marked as an accepted answer). These results provide **another common feature** among these **successful projects** and that is in relation to their community **dealing with StackOverflow questions**. On average, it takes around 9 days for a question to receive an answer that is marked as accepted.

**It can be concluded that each community is different when it comes to its size and average responsitivity for each project, but all are generally reasonably responsive.**
